# CLAUDE.md — Whetstone MCP

> A tool that sharpens AI output by encoding human judgment.

## What Is This?

Whetstone MCP is an MCP server with a SQLite backend that captures, encodes, and scales the human skill of rejection. When a human says "no" to AI output and explains why, that moment creates knowledge — a constraint, a rule, a piece of encoded taste. Today, those moments evaporate. Whetstone makes them persist and compound.

## Requirements

These are firm constraints, not preferences:

- **Standalone repository.** Whetstone is its own project, not embedded in any other codebase.
- **Agent-agnostic.** Must work with any MCP-compatible agent: Claude Code, Codex, Kiro, Cursor, or anything else. No agent-specific assumptions in the core.
- **Per-project storage.** Each project has its own `.whetstone/` directory with its own SQLite database. The MCP server is configured to point at that project's db. The database is committed to the repo so constraints travel with the code.
- **Capture where the work happens.** The MCP protocol means capture is a side effect of the rejection you're already performing, inside the conversation. No context switching.
- **Git-friendly exports.** A pre-push hook exports a human-readable snapshot of all constraints before each push, so constraint changes are visible in diffs alongside the code they govern.

## Tech Stack

| Technology | Purpose | Rationale |
|---|---|---|
| TypeScript | Server language | Widest MCP SDK support, npm-publishable, most agents use Node-based MCP |
| better-sqlite3 | Database | Native SQLite bindings — fast, atomic writes, WAL mode for concurrent access |
| @modelcontextprotocol/sdk | MCP protocol | Standard MCP server implementation |
| ulid | Identifiers | Sortable, unique, no coordination required |

**Why TypeScript over Python?** The MCP ecosystem is most mature in TypeScript. Most agent tools (Claude Code, Cursor) consume MCP servers via `npx` or direct Node execution. Python MCP servers work but add friction (virtual environments, pip installs). TypeScript also aligns with the npm publishing model for easy distribution.

**Why per-project SQLite over a central database?** The database is committed to the repo, so constraints travel with the code and team members get them via git. Each project's taste is self-contained. No cross-project contamination by design.

**Why better-sqlite3 over sql.js?** better-sqlite3 provides atomic writes, WAL mode for concurrent access (dashboard + MCP server), and significantly better performance. It requires a C++ compiler for native compilation, but the target audience is developers who have build tools installed. sql.js (pure WASM) was used initially but its load-entire-file/write-entire-file pattern caused data loss during rapid sequential CLI invocations.

## Getting Started

### 1. Install Whetstone

Whetstone is a private GitHub repository, not published to npm. Choose one of these install methods:

#### Option A: Clone + Link (recommended for development)

Best when you're actively developing whetstone or want easy updates.

```bash
git clone git@github.com:your-org/whetstone.git ~/tools/whetstone
cd ~/tools/whetstone
make setup
```

This installs dependencies, compiles TypeScript, and symlinks both `whetstone` and `whetstone-mcp` globally. To update:

```bash
cd ~/tools/whetstone
git pull && make build
```

No reinstall needed — the symlink picks up the new build automatically.

#### Option B: Global install from GitHub

Simpler setup, but requires reinstalling to update.

```bash
npm install -g --install-links github:frontier-collective/whetstone-mcp
```

To update, run the same install command again.

#### Option C: Direct path reference

Skip the global install entirely and point your MCP config at a local clone (see step 3 below).

### 2. Initialize Whetstone in your project

```bash
cd ~/projects/my-app
whetstone init
```

This creates:
- `.whetstone/whetstone.db` — empty SQLite database
- `.whetstone/exports/` — directory for pre-push snapshots
- `.git/hooks/pre-push` — git hook that exports constraints before each push

Commit the `.whetstone/` directory to your repo.

### 3. Configure your agent

Add Whetstone as an MCP server in your project's agent config. The MCP server starts automatically when the agent starts — you never run it manually.

#### Claude Code

In the project's `.claude/settings.json`:

```json
{
  "mcpServers": {
    "whetstone": {
      "command": "whetstone-mcp",
      "env": {
        "WHETSTONE_DB": ".whetstone/whetstone.db"
      }
    }
  }
}
```

If you didn't install globally, reference the repo directly:

```json
{
  "mcpServers": {
    "whetstone": {
      "command": "node",
      "args": ["/path/to/whetstone/dist/index.js"],
      "env": {
        "WHETSTONE_DB": ".whetstone/whetstone.db"
      }
    }
  }
}
```

The `WHETSTONE_DB` path is relative to the project root. The server creates the database and `.whetstone/` directory if they don't exist.

#### Other Agents

Any MCP-compatible agent can connect using the same pattern — point it at the server entry point with `WHETSTONE_DB` set to the project's `.whetstone/whetstone.db`. The server uses stdio transport (standard MCP).

### 4. Use it

No further setup. The whetstone tools (`reject`, `constrain`, `get_constraints`, etc.) are now available in your agent conversations. When you push, the pre-push hook automatically exports a snapshot of your constraints.

## Releasing

`make release minor` (or `patch` / `major`) handles the full release flow: bump version, generate changelog, create release branch, merge to master and develop, tag.

The changelog generator (`scripts/changelog.mjs`) uses the Claude API to produce polished, grouped release notes when `ANTHROPIC_API_KEY` (or `CLAUDE_API_KEY`) is set in the environment. Without an API key, it falls back to raw commit messages.

```bash
# Preview release notes without writing
ANTHROPIC_API_KEY=sk-... node scripts/changelog.mjs 0.5.0 --dry-run

# Full release (API key optional)
ANTHROPIC_API_KEY=sk-... make release minor
```

The API key is never committed — it's expected in the shell environment at release time only.

## Build Order (MVP)

Build in this order. Each step should be functional before moving to the next.

1. **Project scaffolding** — package.json, tsconfig.json, directory structure
2. **Database layer** — connection management, schema creation, migrations
3. **Core types** — TypeScript interfaces for constraints, rejections
4. **MCP server skeleton** — entry point, stdio transport, tool registration
5. **`reject` tool** — log a rejection (the most common entry point)
6. **`constrain` tool** — create a durable constraint
7. **`get_constraints` tool** — query constraints for proactive application (the highest-value tool)
8. **`search` tool** — free-text search
9. **`applied` tool** — usage tracking
10. **`update_constraint` tool** — refine/deprecate
11. **`export` tool** — markdown/JSON export for graduation
12. **CLI export command** — standalone export for git hook usage
13. **`init` command** — set up `.whetstone/` directory and install pre-push git hook
14. **`patterns` tool** — aggregate rejection analysis
15. **`stats` tool** — usage statistics

Steps 1-7 constitute the MVP. Everything after that is enhancement.

## Database Schema

Key tables:

- **rejections** — raw rejection events (the kindling)
- **constraints** — distilled, durable rules (the blade)

No project scoping — the database is per-project, so all data in it belongs to that project.

## Schema Migrations

The database schema is versioned using SQLite's `PRAGMA user_version`. Migrations run automatically on first `getDb()` call — no user action required.

### How it works

- Migrations are defined in `src/db/migrations.ts` as an ordered array of functions
- Each migration brings the database from version N to version N+1
- `connection.ts` calls `runMigrations(db)` on every new connection
- If the database is already current, it's a no-op (single integer comparison)
- Migration output goes to stderr (stdout is the MCP JSON-RPC channel)

### Adding a new migration

When you change the database schema:

1. **Append** a new function to the `migrations` array in `src/db/migrations.ts`
2. **Never** remove, reorder, or modify existing migrations
3. Use `ALTER TABLE ... ADD COLUMN` for new nullable columns
4. Use `CREATE TABLE` (without `IF NOT EXISTS`) for new tables
5. Use `CREATE INDEX` for new indexes
6. Update types in `src/lib/types.ts` to match the new schema
7. Update relevant tools to use the new columns/tables

### Rules

- Migration 0→1 is special: uses `CREATE TABLE IF NOT EXISTS` for backward compatibility with pre-migration databases
- All subsequent migrations use direct DDL (no `IF NOT EXISTS`)
- Each migration runs in a transaction — failures roll back cleanly
- Foreign keys are disabled during migrations to allow ALTER TABLE operations
- The version is stamped after each successful migration, not at the end

## Key Design Decisions

### Per-Project Database

Each project gets its own `.whetstone/whetstone.db`. The MCP server reads `WHETSTONE_DB` env var to find the database. No project slugs, no cross-project queries, no multi-tenancy. The filesystem boundary is the scoping boundary.

### Constraint Lifecycle

```
Rejection (raw event)
  → Constraint (encoded rule, status: active)
    → Graduated (exported to project docs like CLAUDE.md, steering files)
      → Deprecated in Whetstone (avoid duplication)
```

### Relationship to Project Docs

Many projects already have informal constraint libraries — CLAUDE.md files, steering files, cursor rules, Codex instructions. Whetstone complements these:

- **Whetstone** is the living, queryable, evolving constraint store
- **Project docs** are the graduated, stable, committed rules
- The `export` tool bridges the two: when a constraint has proven durable, export it as markdown and paste it into the appropriate project doc, then deprecate it in Whetstone

This avoids duplication while keeping both systems useful.

## Git Hooks

### Pre-Push Export Hook

Whetstone includes a pre-push git hook that automatically exports a human-readable snapshot of all constraints before each push. This ensures constraint changes are always visible in diffs alongside the code they govern.

`whetstone hook` installs two files:

- `.git/hooks/pre-push-whetstone` — the whetstone export script (always written/updated)
- `.git/hooks/pre-push` — a dispatcher that calls `pre-push-whetstone` (prompts before overwriting an existing hook)

This dispatcher pattern lets whetstone coexist with other pre-push hooks. Add additional sub-hooks to the dispatcher as needed.

**What the hook does:**
1. Exports all active constraints to a temp file
2. Compares against the most recent existing export — skips if unchanged
3. If constraints have changed, moves the export to `.whetstone/exports/<timestamp>.md`
4. Commits the export and aborts the push (git pre-push hooks can't add commits to the in-flight push)
5. User pushes again — the diff check passes instantly and the push goes through with the snapshot included

**What the export file contains:**
- All active constraints grouped by domain
- Each constraint's title, rule, severity, reasoning, and examples
- Rejection count and usage stats

The timestamped files create a history of constraint evolution. Diffs between consecutive exports show exactly what taste was added, refined, or deprecated.

## Coding Conventions

- Strict TypeScript (`strict: true` in tsconfig)
- ES modules
- No classes for tool handlers — plain functions
- Error messages should be human-readable (these surface in agent conversations)
- No over-engineering — this is a focused tool, not a framework
- Do not add Co-Authored-By lines to commit messages

## Commit Messages

Follow Conventional Commits with WHET story code prefixes.

**Format:**

```
[WHET-NNNN] type: short description    # when related to a story
type: short description                 # when not related to a story
```

**Types:** `feat`, `fix`, `chore`, `docs`, `refactor`, `test`, `style`, `ci`, `perf`

**Multi-line commits** (when a commit touches multiple concerns):

```
[WHET-0007] feat: add AI-generated changelog
chore: add .env.example for API key config
```

**Rules:**
- First line under 70 characters
- Use imperative mood ("add", not "added" or "adds")
- WHET code goes first when applicable — only if there's a clear match
- One type prefix per line
- No trailing periods
